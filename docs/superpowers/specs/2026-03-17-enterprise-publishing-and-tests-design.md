# Enterprise-Grade PyPI Publishing & Test Quality

**Date:** 2026-03-17
**Status:** Approved
**Approach:** Two-phase rollout (Phase 1: Infrastructure, Phase 2: Tests)
**Ordering:** Phase 2 depends on Phase 1 — CI gates must exist before test improvements land.

---

## Overview

Upgrade resq-mcp from alpha-quality packaging and testing to enterprise-grade:
OIDC Trusted Publisher for PyPI, Sigstore signing, SLSA provenance, full CI matrix,
rewritten tests with integration coverage and property-based testing.

---

## Prerequisites

- **PyPI Trusted Publisher:** Configure on pypi.org → project settings → Publishing →
  add GitHub as a trusted publisher (repo: `resq-software/mcp`, workflow: `publish.yml`,
  environment: `release`).
- **TestPyPI Trusted Publisher:** Separate configuration on test.pypi.org with the same
  repo but workflow `ci.yml`. Note: TestPyPI uploads are permanent and versions cannot
  be reused — use `.devN` suffixes or commit-hash-based versions for PR builds.
- **Rollback:** Keep `PYPI_TOKEN` secret available during transition. If OIDC fails,
  the publish step can fall back to token auth by uncommenting the `pypi_token` parameter.
  Remove the secret only after 2+ successful OIDC releases.

## Phase 1: Infrastructure & Publishing

### 1.1 — PyPI OIDC Trusted Publisher

Replace `PYPI_TOKEN` secret with OpenID Connect authentication. GitHub Actions
authenticates directly with PyPI via OIDC identity provider.

- Use `pypa/gh-action-pypi-publish` (native OIDC) instead of
  `python-semantic-release/publish-action` for the upload step.
- Keep `python-semantic-release` for version bumping, tagging, and GitHub Release creation.
  Artifacts (SBOM, signatures, provenance) are uploaded to the release it creates.
- Add TestPyPI publish on PRs for pre-merge validation. Version suffix: inject
  `.dev<github.run_number>` by dynamically patching the version in `pyproject.toml`
  before building (e.g., `sed -i "s/^version = .*/version = \"$(cat version).dev${GITHUB_RUN_NUMBER}\"/" pyproject.toml`).
  This is a CI-only mutation, never committed.

**Workflow:**
```
PR → build sdist/wheel → upload to TestPyPI (OIDC)
main → semantic-release (bump + tag + GitHub Release) → build → PyPI (OIDC) → attach artifacts → Docker
```

### 1.2 — Sigstore Signing & SLSA Provenance

- `sigstore/gh-action-sigstore-python` signs built distributions.
- SLSA provenance at **Level 2** (not Level 3 — Level 3 requires the build to run inside
  the SLSA generator's reusable workflow, which conflicts with our multi-job pipeline).
  Use `slsa-framework/slsa-github-generator` with the generic provenance generator
  pointing at the build job's artifacts.
- Signatures + provenance attached to the GitHub Release created by semantic-release.
- `gh attestation verify` smoke test in pipeline.

### 1.3 — SBOM Generation

- `anchore/sbom-action` generates CycloneDX SBOM from built wheel.
- Attached to GitHub Release alongside signed artifacts.

### 1.4 — CI Pipeline Hardening

**New job graph in `ci.yml`:**
```
lint (ruff check + format --check)  ──┐
typecheck (mypy --strict)             ├──► test (matrix: 3.11, 3.12, 3.13, 3.14-dev) ──► build-verify
security (ruff S rules, own gate)     ┘
```

- **Lint:** `ruff check src/ tests/` + `ruff format --check src/ tests/`
- **Typecheck:** `mypy src/` — strict mode is configured in pyproject.toml but never ran
  in CI until now. This is a **net-new CI step**.
- **Security:** Ruff `S` (bandit) rules are already in pyproject.toml — this promotes
  them to a separate CI gate for visibility, not a new rule set.
- **Test matrix:** 3.11, 3.12, 3.13, 3.14-dev (3.14-dev: `continue-on-error: true`)
- **Build verification:** `python -m build` + `twine check dist/*`
- **Coverage gate:** Fail if coverage drops below threshold (set in Phase 2 after
  measuring the baseline — see Section 2.7).

### 1.5 — Packaging Metadata

- Add `py.typed` marker in `src/resq_mcp/`.
- Expand classifiers: Python 3.12, 3.13, topic classifiers. Add 3.14 when it reaches GA.
- Add `project.urls`: Documentation, Changelog, Issue Tracker.
- Keep using `[dependency-groups]` (PEP 735) for dev deps — this is the modern standard.
  Do NOT add duplicate `[project.optional-dependencies]` section.
- `CHANGELOG.md` auto-generated by semantic-release.

### 1.6 — Release Workflow (revised publish.yml)

```
jobs:
  release:        # semantic-release: bump version, create tag, create GitHub Release, generate CHANGELOG
  build:          # build sdist + wheel, verify with twine
  sign:           # Sigstore signing
  provenance:     # SLSA Level 2 provenance generation
  publish-pypi:   # pypa/gh-action-pypi-publish (OIDC)
  publish-github: # upload SBOM, sigs, provenance to the GitHub Release created by semantic-release
  docker:         # build + push tagged Docker image
```

Each job depends on the previous, creating a clear audit trail.
All jobs after `release` are conditional on `released == 'true'` — commits that
don't trigger a version bump (e.g., docs, chore) skip the entire publish chain.

## Phase 2: Test Quality

### 2.1 — New Dev Dependencies

- `aioresponses` — HTTP mocking for aiohttp. Note: no source module currently makes
  HTTP calls (tools return stub data), but aiohttp is a declared dependency and the
  likely HTTP client when stubs are replaced. This is forward-looking infrastructure.
  Also update CLAUDE.md and AGENTS.md to reference `aioresponses` instead of `respx`.
- `hypothesis` — property-based testing
- `pytest-timeout` — prevent hung tests in CI

### 2.2 — Rewrite Weak Tests

**test_server.py** (currently weakest file):
- Background `simulation_processor()` state machine: pending → processing → completed
- SSE `notify_resource_updated()` fires at correct transitions
- `get_deployment_strategy()` tool wrapper end-to-end
- `incident_response_plan()` prompt rendering
- Lifespan context (background task starts/stops)
- `resq://drones/active` resource

**test_validate_incident.py** (currently 2 tests):
- Different validation sources (drone_feed, sensor_network, manual)
- Correlated alert scenarios
- Logging output format for audit trail

**test_telemetry.py** (tests no-ops):
- `trace()` preserves function signatures
- Decorated async functions work correctly
- `setup_telemetry()` logging in debug mode

**test_config.py** (6 tests, solid for Settings — only gap is `validate_environment()`):
- `validate_environment(require_api_key=True)` raises ConfigurationError
- `validate_environment(require_api_key=False)` passes with default
- `.env` file loading

**test_security.py** (6 tests, adequate for core paths — low priority):
- Specific gaps: empty Bearer value (`"Bearer "`), multi-space separator between
  scheme and token, token containing unicode characters
- Existing tests already cover: missing header, invalid scheme, wrong token, valid
  token, case-insensitive Bearer

**test_dtsop.py, test_hce.py, test_pdie.py** — existing unit tests are solid.
Improvements come via edge cases (Section 2.3) and integration flows (Section 2.4),
not rewrites.

### 2.3 — Edge Case Tests

Across all modules:
- Empty string inputs for sector_id, incident_id, drone_id
- Boundary floats: 0.0, 1.0, negative values for confidence
- Model serialization round-trips: `model_dump()` → `Model(**data)`
- Invalid enum values rejected
- Datetime timezone handling: naive vs aware

### 2.4 — Integration Tests (`tests/test_integration.py`)

**Flow 1: Incident → Validation → Strategy → Mission**
```
IncidentReport → hce.validate_incident() → dtsop.get_optimization_strategy()
→ hce.update_mission_params()
```

**Flow 2: Surveillance → Alert → Deployment**
```
tools.scan_current_sector() → pdie.get_predictive_alerts()
→ tools.request_drone_deployment()
```

**Flow 3: Simulation Lifecycle**
```
server.run_simulation() → poll resq://simulations/{id}
→ state transitions → get_deployment_strategy()
```

### 2.5 — Property-Based Testing (Hypothesis)

**Models:**
- Generate random valid Coordinates (lat -90..90, lon -180..180), verify construction
- Fuzz IncidentReport fields, verify serialization round-trips
- Fuzz DeploymentRequest priority — valid enums pass, else ValidationError

**Probabilistic Functions:**
- `scan_current_sector()` N runs — disaster rate ≈ 30% within tolerance
- `get_predictive_alerts()` — probabilities always in [0, 1]

### 2.6 — aioresponses HTTP Mocking Infrastructure

**conftest.py improvements:**
- `aioresponses` fixture for mocking aiohttp external API calls
- Factory fixtures: `make_incident_report()`, `make_simulation_request()`,
  `make_sector_analysis()`
- FastMCP test client fixture for protocol-layer testing
- Pattern established for when stubs are replaced with real HTTP calls

### 2.7 — Coverage Enforcement

- **Step 1:** Run `pytest --cov=src --cov-report=term` to measure current baseline.
- **Step 2:** Set `--cov-fail-under` to the measured baseline (floor, not aspirational).
- **Step 3:** After all Phase 2 tests land, raise the threshold to the new baseline
  (expected ~85%+).
- HTML coverage report as CI artifact.
- Codecov diff check on PRs (integration already exists).
